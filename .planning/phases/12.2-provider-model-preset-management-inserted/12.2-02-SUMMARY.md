# Phase 12.2 Plan 02: API Endpoints, Schema Extensions & LLM Resolution Summary

**Rewrote resolve_langchain_llm for DB-backed sdk_type/base_url dispatch, added 3 model CRUD endpoints via ModelPricing FK join, preset delete protection, extended all response schemas with preset/base_url fields**

## Accomplishments

- Rewrote `resolve_langchain_llm` to read `sdk_type` and `base_url` from AIProviderConfig DB instead of hardcoded `if provider_name == "gemini"` chains
- Added `_get_provider_config` helper method to ProviderManager for system-level config lookup
- Extended `ProviderConfigResponse` with 6 new fields: `description`, `icon`, `sdk_type`, `default_base_url`, `base_url`, `is_preset`
- Created `ProviderModelResponse`, `ModelPricingBrief`, `ModelCreateRequest`, `ModelUpdateRequest` schemas
- Added 3 new endpoints: GET/POST `/{provider_id}/models`, PATCH `/{provider_id}/models/{model_id}`
- Added preset provider delete protection (403 for `is_preset=True`)
- Updated `list_models` to use ModelPricing FK join for provider name resolution
- Extended `PricingCreate`/`PricingResponse` with `provider_config_id`/`model_config_id` FK fields

## Decisions Made

- resolve_langchain_llm dispatches on `sdk_type` (native vs openai_compatible) instead of provider_name string matching — extensible for future providers
- effective_base_url = `config.base_url or config.default_base_url` — user override takes precedence over preset default
- `_get_provider_config` queries system-level config only (`owner_type=system`) — team/personal configs don't affect LLM class selection
