# Phase 12.2: Provider & Model Preset Management - Research

**Researched:** 2026-04-03
**Domain:** Backend data model refactoring + Admin UI restructuring (SQLAlchemy, FastAPI, React/Next.js)
**Confidence:** HIGH

## Summary

Phase 12.2 transforms the provider management system from user-manual-creation to system-preset-plus-user-key-fill mode. The core change is adding `is_preset`, `seed_version`, `description`, `icon`, `default_base_url`, and `sdk_type` fields to `AIProviderConfig`, extending `AIModelConfig` similarly, making `ModelPricing` double as the Provider-Model association table (replacing `AIModelProviderMapping`), and restructuring the Admin UI to show preset provider cards in "ÕĘ▓ķģŹńĮ«"/"ÕŠģķģŹńĮ«" groups with embedded model lists.

The codebase is well-positioned: each provider implementation already has `_KNOWN_MODELS` dictionaries with rich metadata (display_name, capabilities, token limits, etc.), `seed_providers_from_env()` provides the seeding pattern, and the `ProviderCard` component already supports expand/collapse. The major work is: (1) extending DB models + seed logic, (2) rewriting `resolve_langchain_llm` to read `base_url` from DB, (3) restructuring the frontend Provider page + card + modal.

**Primary recommendation:** Treat provider implementation classes as the single source of truth for model metadata. Extend `_KNOWN_MODELS` with pricing and SDK info, then have a centralized seed function read all providers and write to DB with `seed_version` gating.

**When to use:** For any provider that should be preset-seeded.

### Pattern 2: Seed Version Gating

**What:** Each DB row has a `seed_version` field. On startup, the seed function compares the code-declared version with DB. Only processes records where code version > DB version.
**When to use:** Preset Provider/Model/Pricing seeding.

```python
async def seed_preset_providers():
    """Seed preset providers from code declarations. Respects seed_version."""
    CURRENT_SEED_VERSION = 1  # Bump when adding/changing presets

    for meta in get_all_provider_metadata():
        existing = await get_provider_by_name(meta["provider_name"])
        if existing and existing.seed_version >= CURRENT_SEED_VERSION:
            continue  # Already up to date, don't overwrite user changes
        if existing:
            # Version bump: update description/icon but NOT user-edited fields
            existing.seed_version = CURRENT_SEED_VERSION
            existing.description = meta["description"]
            existing.icon = meta["icon"]
        else:
            # New provider
            create_new_preset_provider(meta, seed_version=CURRENT_SEED_VERSION)
```

### Pattern 3: ModelPricing as Association Table

**What:** `ModelPricing` gains FK fields to serve as both pricing data and Provider-Model association.
**When to use:** Replacing `AIModelProviderMapping`.

```python
class ModelPricing(Base):
    # Existing text fields (backward compat)
    provider: Mapped[str]  # "gemini", "openai", etc.
    model: Mapped[str]     # "gemini-2.5-flash", etc.

    # New FK fields
    provider_config_id: Mapped[str | None] = mapped_column(
        String(36), ForeignKey("ai_provider_configs.id"), nullable=True, index=True

    model_config_id: Mapped[str | None] = mapped_column(
        String(36), ForeignKey("ai_model_configs.id"), nullable=True, index=True

    # Relationships
    provider_config = relationship("AIProviderConfig")
    model_config = relationship("AIModelConfig")
```

### Pattern 4: resolve_langchain_llm DB-backed base_url

**What:** `resolve_langchain_llm` reads `base_url` from DB (`AIProviderConfig.base_url` override ŌåÆ `AIProviderConfig.default_base_url` fallback) instead of hardcoding.
**When to use:** Every LangChain LLM construction.

```python
async def resolve_langchain_llm(self, provider_name, model_name, **kwargs):
    api_key, _owner, key_id = await self._resolve_key(provider_name, ...)

    # Read provider config for base_url + sdk_type
    config = await self._get_provider_config(provider_name)
    base_url = config.base_url or config.default_base_url  # Admin override ŌåÆ default

    if config.sdk_type == "native":
        # Gemini uses its own SDK
        from langchain_google_genai import ChatGoogleGenerativeAI
        return ChatGoogleGenerativeAI(model=model_name, api_key=api_key, streaming=True, **kwargs)
    else:
        # openai_compatible: OpenAI, DeepSeek, any OpenAI-compatible
        from langchain_openai import ChatOpenAI
        extra = {"base_url": base_url} if base_url else {}
        return ChatOpenAI(model=model_name, api_key=api_key, streaming=True, **extra, **kwargs)
```

### Anti-Patterns to Avoid
- **Hardcoding provider logic in if/elif chains:** The current `resolve_langchain_llm` uses `if provider_name == "gemini" ... elif provider_name in ("openai", "deepseek")`. This should use `sdk_type` from DB instead, making it extensible.
- **Overwriting user edits on seed:** Never seed over records where the user has made modifications. Use `seed_version` comparison only.
- **Frontend creating providers:** D-01 explicitly locks this ŌĆö remove "Add Provider" button entirely. Providers come from code registration only.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Provider-Model association | Custom join table | `ModelPricing` with FK fields | D-13: dual-purpose table, avoids maintaining two association tables |
| Schema migration | Alembic migration files | `metadata.create_all` + `_auto_migrate_columns` | Project pattern: auto-migrate in dev, no migration files |
| Icon rendering | Upload/store icon images | String key ŌåÆ frontend SVG component mapping | Simpler; finite set of preset providers; no file storage needed |
| Seed idempotency | Custom diffing logic | `seed_version` integer comparison | D-03: simple, deterministic, incrementable |

## Common Pitfalls

### Pitfall 1: Breaking Existing Billing Queries
**What goes wrong:** Billing CRUD and usage-stats queries use `ModelPricing.provider` (text) and `ModelPricing.model` (text) for filtering/grouping. Adding FK fields and changing queries could break existing pricing page.
**Why it happens:** D-13 says "ÕÄ¤ provider / model µ¢ćµ£¼ÕŁŚµ«Ąõ┐ØńĢÖÕÉæÕÉÄÕģ╝Õ«╣’╝īµ¢░Ķ«░ÕĮĢÕÉīµŚČÕåÖõĖżÕźŚ" ŌĆö but API consumers may not be updated simultaneously.
**How to avoid:** Phase the migration: (1) Add FK columns as nullable, (2) new seed records write both text + FK, (3) existing billing CRUD continues using text fields, (4) only new provider-card model list queries use FK joins. Text ŌåÆ FK query migration happens later (Claude's discretion).
**Warning signs:** Billing pricing page shows empty/broken data after deployment.

### Pitfall 2: Seed Function Ordering
**What goes wrong:** Models reference `provider_config_id` via FK. If models are seeded before providers, FK constraint fails.
**Why it happens:** `ModelPricing` FK ŌåÆ `ai_provider_configs.id` and `ai_model_configs.id`.
**How to avoid:** Seed in order: (1) `AIProviderConfig` presets, (2) `AIModelConfig` presets, (3) `ModelPricing` association+pricing records. Use `await session.flush()` between steps.
**Warning signs:** IntegrityError on startup.

### Pitfall 3: Auto-Migrate Column Additions on Existing Tables
**What goes wrong:** `metadata.create_all` does NOT add columns to existing tables. New fields on `AIProviderConfig` / `AIModelConfig` / `ModelPricing` won't appear.
**Why it happens:** SQLAlchemy `create_all` only creates missing tables, not missing columns.
**How to avoid:** Add new columns to the `_auto_migrate_columns()` migration list in `database.py`. This is the established project pattern (see existing entries for `ai_provider_keys.key_hint` and `agent_messages.langchain_messages_json`).
**Warning signs:** `OperationalError: no such column` at runtime.

### Pitfall 4: Delete Mapping Table While It Still Has Import References
**What goes wrong:** Removing `AIModelProviderMapping` class but leaving imports in `ai_providers.py`, `database.py`, etc. causes ImportError.
**Why it happens:** The class is imported in at least 3 files: `database.py` (init_db), `ai_providers.py` (list_models endpoint), `ai_provider_config.py` (model file).
**How to avoid:** Search all imports before deletion: `database.py` line 85, `ai_providers.py` lines 14/416, and any other references. Update all simultaneously.
**Warning signs:** ImportError on startup.

### Pitfall 5: Frontend Provider Card Overloaded
**What goes wrong:** Adding model list, pricing display, add-model form to the already-large ProviderCard (917 lines) makes it unmaintainable.
**Why it happens:** ProviderCard already handles key management, health display, toggle switches, error history, sparkline.
**How to avoid:** Extract the model list into a separate `ProviderModelList` component. The card renders it in the expanded section below the key table.
**Warning signs:** Component exceeds 1200+ lines, hard to reason about state.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Manual provider creation via "Add Provider" | System-preset providers + key fill | This phase | Eliminates misconfiguration; "Dify-style" UX |
| `AIModelProviderMapping` join table | `ModelPricing` as dual-purpose association+pricing | This phase | Simpler schema; one table for both concerns |
| Hardcoded `base_url` in `resolve_langchain_llm` | DB-backed `base_url` with default fallback | This phase | Admin-configurable; extensible to new OpenAI-compatible providers |

## Open Questions

1. **Existing ModelPricing rows without FK**
   - What we know: Current `ModelPricing` rows use text `provider`/`model` fields. D-13 says keep both, new records write both.
   - What's unclear: Should existing rows be backfilled with FK values during seed? Or left as text-only?
   - Recommendation: Backfill in seed function ŌĆö match by `provider` + `model` text to find corresponding `provider_config_id` + `model_config_id`. Low risk since we can match deterministically.

2. **Model uniqueness scope**
   - What we know: `AIModelConfig.model_name` has a unique constraint currently.
   - What's unclear: If two providers can offer the same model (e.g., "gpt-4o" via OpenAI direct and via a proxy), does this break uniqueness?
   - Recommendation: Keep unique for now. Each model name maps to exactly one `AIModelConfig`. `ModelPricing` handles the provider association (one model can have pricing entries for multiple providers).

3. **Icon rendering strategy**
   - What we know: D-01 says only code-registered providers exist (Gemini/OpenAI/DeepSeek).
   - Claude's discretion: Use a string key (e.g., "gemini", "openai", "deepseek") mapped to inline SVG components in the frontend. Simple, no file storage, finite set.
   - Recommendation: Create a `ProviderIcon` component with a switch/map on provider name. Keeps it extensible while avoiding image storage.
