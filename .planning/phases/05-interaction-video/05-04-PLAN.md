Per pitfall 1 (infinite loop): `onChange` callback MUST be wrapped in `useCallback` with stable deps.

State:
```typescript
type BatchState = "idle" | "executing" | "complete";
interface BatchExecution {
  selectedNodes: Node[];
  batchState: BatchState;
  batchId: string | null;
  layers: string[][] | null;
  completedCount: number;
  totalCount: number;
}
```

Hook implementation:
- `useOnSelectionChange({ onChange })` where onChange is `useCallback(({ nodes }) => setSelectedNodes(nodes), [])`
- `canBatchExecute = selectedNodes.length > 1`
- `executeBatch`: POST canvasApi.batchExecute with canvas_id from store + selected node IDs → get batch_id + layers
  - Set batchState = "executing", start polling
  - Use each node's existing `useNodeExecution.execute()` per layer — OR simpler: just call `skillsApi.invoke` per node in topological order from layers
  - Actually, the simpler approach: frontend dispatches execution per node using existing per-node mechanism. The batch orchestration is:
    1. Call `canvasApi.batchExecute` → get `{ batch_id, layers }`
    2. For layer 0 nodes: trigger execution (mark as queued in ReactFlow state)
    3. Poll `canvasApi.batchStatus(batch_id)` every 3s
    4. When all layer N nodes complete, layer N+1 nodes become executable
    5. Frontend updates node data via `setNodes()` for status display
- `clearSelection`: resets selectedNodes to [], deselects in ReactFlow
- `dismissBar`: sets batchState back to "idle"

Poll logic:
```typescript
const pollRef = useRef<ReturnType<typeof setInterval> | null>(null);
```
- Start poll on batchState === "executing"
- Each poll: GET batchStatus → update completedCount from node_statuses
- When all nodes completed/failed → batchState = "complete", show for 2s, then "idle"

Return: `{ selectedNodes, canBatchExecute, batchState, executeBatch, clearSelection, dismissBar, completedCount, totalCount }`
