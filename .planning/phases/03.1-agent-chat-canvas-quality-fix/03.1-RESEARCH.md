## Summary

Phase 03.1 is a pure bug-fix / quality phase that touches 12 specific issues across two subsystems: Agent Chat (backend + frontend) and Canvas (frontend-only + minimal backend writeback). No new features, no new libraries, no new architecture — all work uses existing code patterns and APIs.

The Agent Chat fixes are straightforward wiring: mount the already-implemented `get_pipeline_toolset()`, pass project/canvas query results into `build_system_prompt()`, fix stale closure in `useCallback`, and call the already-existing `agentApi.getMessages` on session switch. The Canvas fixes require extracting the upstream-data-from-edges pattern (already working in `extract-node.tsx`) into a reusable `useUpstreamData` hook, adding debounced node persistence using the OpenLovart `isSavingRef`/`needsSaveRef` pattern, syncing edge deletion to backend, and writing skill execution results back to `CanvasNode`.

**Primary recommendation:** Treat this as 12 discrete, well-scoped patches. Every API and data model needed already exists — this is pure wiring/integration work with zero schema changes.

## Architecture Patterns

### Pattern 1: Upstream Data Hook (useUpstreamData)

**What:** Extract the edge-traversal + data-aggregation logic from `extract-node.tsx` into a reusable hook that all execution nodes consume.

**Current state in extract-node.tsx (lines 62-70):**
```typescript
const inputText = useMemo(() => {
  if (nodeData.text) return nodeData.text as string;
  const incoming = edges.filter((e) => e.target === id);
  for (const edge of incoming) {
    const sourceNode = nodes.find((n) => n.id === edge.source);
    if (sourceNode?.data?.text) return sourceNode.data.text as string;
  }
  return "";
}, [id, nodeData.text, edges, nodes]);
```

**Target hook API (D-07/D-08/D-09):**
```typescript
interface UpstreamData {
  text: string[];
  json: object[];
  imageUrl: string[];
}

function useUpstreamData(nodeId: string): UpstreamData {
  const edges = useEdges();
  const nodes = useNodes();
  return useMemo(() => {
    const incoming = edges.filter((e) => e.target === nodeId);
    const result: UpstreamData = { text: [], json: [], imageUrl: [] };
    for (const edge of incoming) {
      const src = nodes.find((n) => n.id === edge.source);
      if (!src) continue;
      const d = src.data as Record<string, unknown>;
      // Text from text-input nodes or LLM result_text
      if (typeof d.text === "string" && d.text) result.text.push(d.text);
      if (typeof d.result_text === "string" && d.result_text) result.text.push(d.result_text);
      // JSON from extract results
      if (d.result_data && typeof d.result_data === "object") result.json.push(d.result_data as object);
      // Image URLs from image-gen results
      if (typeof d.result_url === "string" && d.result_url) result.imageUrl.push(d.result_url);
    }
    return result;
  }, [nodeId, edges, nodes]);
}
```

**Key decisions:**
- Data classification uses `connection-rules.ts` NODE_IO types as source of truth
- Multi-upstream same-type inputs aggregate into arrays (D-08)
- Single-upstream convenience: node code can do `upstream.text[0] ?? ""` for first-item access
- Source node data mapping: `text-input` → text in `data.text`; `llm-generate` → text in `data.result_text` or execution result; `extract` → json in execution `data`; `image-gen` → imageUrl in `data.result_url`

### Pattern 2: Node Persistence with Debounce + Concurrency Control (D-11/D-12)

**What:** Per-node auto-save hook that debounces text/config changes to `canvasApi.updateNode`, with `isSavingRef`/`needsSaveRef` to prevent concurrent writes.

**OpenLovart reference pattern:**
```typescript
const isSavingRef = useRef(false);
const needsSaveRef = useRef(false);

const saveNode = useCallback(async (nodeId, data) => {
  if (isSavingRef.current) {
    needsSaveRef.current = true;
    return;
  }
  isSavingRef.current = true;
  try {
    await canvasApi.updateNode(nodeId, data);
  } finally {
    isSavingRef.current = false;
    if (needsSaveRef.current) {
      needsSaveRef.current = false;
      saveNode(nodeId, data);
    }
  }
}, [...]);
```

**Canvex adaptation:**
- OpenLovart does full-canvas save; Canvex does per-node `updateNode(nodeId, { config })` — incremental, not full-replace
- Three triggers per D-11: (1) 2s debounce on text/config change, (2) immediate on blur, (3) immediate on execute/connect/delete
- Hook name: `useNodePersistence(nodeId, data)` — returns `{ save, isSaving }`

### Pattern 4: SSE Heartbeat Fix

**Current dead code (agent.py lines 160-164):**
```python
async def _heartbeat():
    while not heartbeat_stop.is_set():
        await asyncio.sleep(15)
        if not heartbeat_stop.is_set():
            yield sse_heartbeat()
```

**Problem:** `_heartbeat` is an async generator (`yield`), but it's never iterated. It was meant to be an `asyncio.Task` that periodically sends heartbeat events. However, `EventSourceResponse(ping=15)` from `sse-starlette` already handles keep-alive pings at the transport level.

**Fix:** Remove the dead `_heartbeat` function and related `heartbeat_task`/`heartbeat_stop` variables. The `ping=15` parameter on `EventSourceResponse` (line 327) already provides keep-alive functionality. Clean removal is safe.

- **Don't use localStorage for chat messages** — Phase 03 decision: "No persist on chat store — session-scoped, not localStorage"

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Upstream data aggregation | Per-node inline useMemo | Shared `useUpstreamData` hook | 3 nodes need same logic; extract-node already has working impl |
| Debounce timer | Manual setTimeout management | `useRef` timer + `isSavingRef` pattern | OpenLovart pattern proven; handles concurrent save edge cases |
| SSE keepalive | Custom heartbeat generator | `EventSourceResponse(ping=15)` | sse-starlette built-in; current heartbeat code is dead code |
| Agent context query | Inline SQL in chat endpoint | SQLAlchemy `selectinload` + service method | Follows existing codebase patterns |
| Edge deletion sync | Custom onEdgesChange handler | ReactFlow `onEdgesChange` with `remove` filter | ReactFlow provides change type metadata |

## Common Pitfalls

### Pitfall 1: useCallback with Empty Deps (Issue #5)
**What goes wrong:** `sendMessage` closes over `projectId`/`canvasId` at mount time. If they change later, the function sends stale values.
**Why it happens:** `useCallback(async (content) => { ... }, [])` — deps array is empty.
**How to avoid:** Use `useChatStore.getState()` inside the callback (already partially done on line 53-54) to always read fresh state. Since the hook already reads from store inside the callback, the fix is mainly ensuring ALL state reads go through `getState()` rather than closure. The current code does `const { projectId, canvasId, ... } = store;` on line 55 which captures at call time — this is actually fine since `getState()` returns current snapshot. The real fix is that `sendMessage` itself is fine (it calls `getState()` each invocation), but `useCallback(fn, [])` means React never recreates it, which is correct for this pattern.
**Warning signs:** Chat always sends to the first project/canvas opened.

**Actually upon re-examination:** The code DOES call `useChatStore.getState()` inside the callback (line 53). So `projectId`/`canvasId` ARE fresh each call. The `[]` deps is actually the correct pattern when using `getState()`. BUT: the `setSession` call on line 76 reads store state that could be stale if multiple rapid calls happen. The real stale closure risk is minimal — the concern may be overstated. Still, let the planner decide if a refactor is warranted.

### Pitfall 2: Edge Deletion Race Condition (Issue #11)
**What goes wrong:** User deletes edge in ReactFlow UI → `onEdgesChange` fires with `remove` type → but no API call to `canvasApi.deleteEdge(edgeId)`.
**Why it happens:** `onEdgesChange` is ReactFlow's built-in handler; it updates local state only. `handleConnect` correctly calls `canvasApi.createEdge`, but there's no matching deletion path.
**How to avoid:** Intercept `onEdgesChange`, filter for `type === "remove"`, call `canvasApi.deleteEdge(change.id)` for each removed edge. Must handle the case where edge IDs from ReactFlow differ from backend IDs (they shouldn't — `toFlowEdge` preserves backend `e.id`).
**Warning signs:** Deleted edges reappear on refresh.

### Pitfall 3: Node Data Shape Inconsistency
**What goes wrong:** Different nodes store execution results in different `data` fields. `text-input` uses `data.text`, `llm-node` reads `execution.data.text`, `image-gen` reads `execution.data.url`. The `useUpstreamData` hook must know where each node type stores its output.
**Why it happens:** Phase 02 nodes were built independently without a shared data contract.
**How to avoid:** Define a canonical output data shape: after execution completes, write results back to node `data` in a standardized way — `data.result_text` for text output, `data.result_url` for image output, `data.result_data` for JSON output. The `useUpstreamData` hook reads from these canonical fields.
**Warning signs:** Upstream data is always empty even when source node has completed.

### Pitfall 4: Suggestion Chips Only Add Bubble (Issue #4)
**What goes wrong:** Clicking a suggestion chip only adds a user message bubble to the UI — never sends it to the backend.
**Why it happens:** `handleSuggestionSend` in `chat-messages.tsx` (line 67-77) only calls `addMessage()`, never calls `sendMessage()` from the chat hook.
**How to avoid:** The `ChatMessages` component needs access to `sendMessage` from `useAgentChat()`. Either: (a) lift `sendMessage` to a parent and pass as prop, or (b) call `useAgentChat()` in `ChatMessages`. Option (a) is cleaner — parent already has the hook.
**Warning signs:** Clicking chips shows user bubble but no AI response.

### Pitfall 5: Session Switch Doesn't Load History (Issue #3)
**What goes wrong:** Switching to a different session clears messages but doesn't load that session's history from backend.
**Why it happens:** `setSession` in `chat-store.ts` (line 62) does `set({ sessionId, messages: [] })` — clears messages but doesn't trigger any API call.
**How to avoid:** After `setSession(id)`, call `agentApi.getMessages(id)` and populate messages. This should happen in the component/hook that manages session switching, not in the store itself. The `agentApi.getMessages` endpoint already exists and works (tested in `test_agent_api.py`).
**Warning signs:** Switching sessions always shows empty chat.

## Open Questions

1. **Context tool registration mechanism**
   - What we know: D-02 says register tools like `get_script_content`, `get_character_details`, etc. These need DB access.
   - What's unclear: Should these be a third `FunctionToolset` alongside `SkillToolset` and `PipelineToolset`, or registered as new skills in `SkillRegistry`?
   - Recommendation: Use `FunctionToolset` (like pipeline_tools.py pattern) — these are simple DB queries, not AI skills. Keeps skill registry clean.

2. **Node data field normalization**
   - What we know: Different nodes store output in different fields (`text`, `result_text`, `data`, `result_url`)
   - What's unclear: Should we normalize all nodes to a single output convention, or let `useUpstreamData` handle the mapping?
   - Recommendation: Normalize — after execution, write to canonical fields (`result_text`, `result_url`, `result_data`). The `useUpstreamData` hook reads these. Text-input is special: its "output" is `data.text` (user-entered), not an execution result.

3. **Backend `result_url` not in `NodeUpdate` schema**
   - What we know: `NodeUpdate` schema has `result_text` but NOT `result_url` (see schemas/canvas.py line 37-44)
   - What's unclear: Was this intentional?
   - Recommendation: Add `result_url: str | None = None` to `NodeUpdate` schema. The model field exists (`CanvasNode.result_url`), it just wasn't exposed in the update schema. Also add `error_message: str | None = None` for completeness.
