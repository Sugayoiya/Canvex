# Phase 03.1: Agent Chat + Canvas Quality Fix - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

修复 Phase 03 交付的 Agent Chat 和 Canvas 中的 12 个质量问题，使核心功能链路达到可用状态。不新增功能，不做后端拓扑执行（Graph-Based 留给 Phase 05）。

### 问题清单

**Agent Chat（6 项）：**
1. Pipeline 工具未接入聊天端点（只有 SkillToolset，缺 `get_pipeline_toolset()`）
2. 项目/画布上下文未注入 Agent（system prompt 中占位为空）
3. 切换会话不加载历史消息（`setSession` 清空 messages，不调 `getMessages`）
4. 建议 chips 点击不发送（`handleSuggestionSend` 只加气泡，未调 `sendMessage`）
5. `sendMessage` 闭包过期（`useCallback` 依赖数组为 `[]`，projectId/canvasId 陈旧）
6. 心跳代码是死代码（`_heartbeat` 从未启动）

**Canvas（6 项）：**
7. LLM/Image-gen 节点不从上游边获取输入（只有 Extract 有边聚合逻辑）
8. 无整图执行编排（本次仅修前端数据流，不加后端拓扑执行）
9. 节点文本/配置不持久化（text-input 等修改只在前端内存，刷新丢失）
10. Output 节点是静态的（不自动汇聚上游数据）
11. 删边不同步后端（前端删边不调 `canvasApi.deleteEdge`）
12. 技能执行结果不回写节点（SkillExecutor 不更新 `CanvasNode.status/result`）

</domain>

<decisions>
## Implementation Decisions

### Agent 上下文注入
- **D-01:** 采用按需加载策略 — 先注入项目/画布结构摘要（项目名、画布名、节点类型/数量、角色/场景列表名称），Agent 需要详情时通过工具调用获取
- **D-02:** 新增上下文查询工具注册到 Agent：`get_script_content`、`get_character_details`、`get_scene_details`、`get_canvas_state` 等，让 Agent 按需拉取完整内容
- **D-03:** `context_builder.build_system_prompt()` 接收结构化摘要参数，`agent.py` 的 `chat` 端点查询项目/画布数据后传入

### Agent 思考范式
- **D-04:** 分级响应策略 — 简单/明确请求（"提取角色"）直接调工具执行；复杂/模糊请求（"帮我完善这个剧集"）先分析并输出执行计划，用户确认后再执行
- **D-05:** Pipeline 工具必须接入聊天端点，`toolsets=[SkillToolset(...), get_pipeline_toolset()]`，让 Agent 对完整流程请求能调用 Pipeline
- **D-06:** Agent 语言风格不做硬性约束，交由模型自行把握

### Canvas 节点数据流
- **D-07:** 类型化数据流 — 节点间传递 text / json / imageUrl 三种类型，连线时校验类型兼容性（复用已有 `connection-rules.ts`）
- **D-08:** 多上游同类型聚合为数组 — 节点收到多个上游 text 输入时得到 `text: string[]`，节点内部决定如何消费（拼接、逐个处理、取第一个等）
- **D-09:** 每个节点声明输入/输出端口类型 — 抽取统一的 `useUpstreamData` hook，替代 Extract 节点里的内联 `useMemo` 逻辑，所有执行类节点复用
- **D-10:** Output 节点自动汇聚上游数据并按类型展示（text 渲染文本、imageUrl 渲染图片、json 渲染格式化 JSON）

### 节点持久化时机
- **D-11:** 混合三层持久化策略：
  - 编辑中：防抖 2 秒无操作自动保存节点 config/text 到后端
  - 失焦时：blur 立即保存当前节点
  - 关键操作：执行技能、连线/删边、节点删除 → 立即持久化
- **D-12:** 参考 OpenLovart 的 `isSavingRef` + `needsSaveRef` 并发控制模式，避免并发写入冲突
- **D-13:** 技能执行结果回写：SkillExecutor 完成后更新 `CanvasNode.status`、`result_text`、`result_url`（后端），前端通过 poll 结果同步更新节点 `data`

### 整图执行边界
- **D-14:** 本次仅修前端数据流 — 节点间能通过边传递类型化数据，但每个节点仍需手动点击执行
- **D-15:** 不新增后端拓扑执行 API，Graph-Based 执行留给 Phase 05（与 Phase 03 D3 一致）

### Claude's Discretion
- Agent 语言风格和回复详略程度
- 防抖保存的具体实现细节（精确延迟值、重试逻辑）
- `useUpstreamData` hook 的具体 API 设计
- 上下文查询工具的具体返回格式
- 心跳清理的具体实现方式

</decisions>

- `connection-rules.ts` 的类型兼容矩阵 — 扩展支持 text[] / imageUrl[] 聚合

- `agent_service.py` 的 `create_agent(project_name, canvas_name)` — 接口已预留，需在 chat 端点传入

### Established Patterns
- SSE 阶段机（thinking → token → tool_call → tool_result → done）— 不变
- `canvasApi.updateNode` / `createEdge` / `deleteEdge` — 后端 CRUD API 已就绪
- `skillsApi.invoke` + `skillsApi.poll` — 前端技能调用模式不变
- OpenLovart 的 `isSavingRef` + `needsSaveRef` 并发控制 — 可直接复用

### Integration Points
- `agent.py` chat 端点：挂载第二个 toolset + 查询项目/画布数据
- `canvas-workspace.tsx` `onEdgesChange`：拦截 remove 类型调 `canvasApi.deleteEdge`
- `chat-store.ts` `setSession`：触发 `agentApi.getMessages` 加载历史
- `use-node-execution.ts` execute 完成后：调 `canvasApi.updateNode` 回写结果

<specifics>
## Specific Ideas

- 参考 OpenLovart 的防抖保存模式（2s debounce + isSavingRef 并发控制），但 Canvex 采用增量更新（per-node `updateNode`）而非全量替换
- 上游数据 hook 应返回 `{ text: string[], json: object[], imageUrl: string[] }` 结构，节点按需消费
- Agent 上下文查询工具应利用现有 `SkillRegistry` 注册机制，作为新的 skill category（如 `context.*`）

</specifics>
