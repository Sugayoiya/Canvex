# Cross-AI Plan Review — Phase 12.1

## Codex Review

### PLAN 01: LangChain `@tool` Pure Function Toolkit

**Summary**
Plan 01 is a solid base for tool normalization and removes a major class of hidden reasoning-in-tools bugs. The main risk is not implementation complexity, but runtime behavior: tool count, context propagation reliability, and blocking side effects in "AI tools."

**Strengths**
- Clear separation of read/write/AI/skill tools.
- Correct choice to use per-call `AsyncSessionLocal` instead of shared DB sessions.
- Explicit path traversal guard for resource reads.
- Context propagation is planned, not ad hoc.

**Concerns**
- **HIGH**: Returning JSON strings from all tools can degrade tool-call reliability versus typed structured outputs (the model must re-parse text).
- **HIGH**: 17 tools is near the known reliability cliff; no tool-selection/routing strategy is defined.
- **MEDIUM**: `contextvars` can be lost across some async boundaries/background execution unless carefully set at entrypoints.
- **MEDIUM**: AI tools may block event loop if image/video operations are synchronous or long-running.
- **MEDIUM**: No explicit timeout/retry/circuit-breaker contract per tool.

**Suggestions**
- Return typed dict/Pydantic-shaped outputs; stringify only at API boundary.
- Add tool gating middleware (context-aware subset of tools per request).
- Define standard tool wrapper: timeout, retry policy, structured error envelope.
- Force long-running AI tools onto async job queue with immediate job-id return.
- Add authz checks in write tools (team/project/user scope) as first-class behavior.

**Risk Assessment:** MEDIUM

### PLAN 02: Anthropic `SKILL.md` + SkillLoader

**Summary**
The plan aligns with skill-based architecture and progressive disclosure goals. Biggest risks are spec drift, cache invalidation, and safety boundaries for skill/resource loading.

**Strengths**
- Three-level loading model is well aligned with context-efficiency goals.
- Metadata cache for L1 is practical for startup performance.
- Migration from Python reasoning handlers to declarative skill docs is directionally correct.
- `yaml.safe_load` is the right parser choice.

**Concerns**
- **HIGH**: No explicit cache invalidation/hot-reload strategy for `SKILL.md` updates.
- **MEDIUM**: Skill frontmatter assumptions may be stricter than current spec requires (risk of unnecessary validation failures).
- **MEDIUM**: Meta-skill orchestration (`episode-pipeline`) can create recursive/verbose instruction chains without guardrails.
- **MEDIUM**: L3 resource loading needs strict path allowlist and size caps, not just traversal checks.
- **LOW**: No fallback behavior described for malformed frontmatter/body.

**Suggestions**
- Add mtime/hash-based cache invalidation with safe reload path.
- Treat frontmatter schema as tolerant (warn + skip invalid fields, don't hard-fail all skills).
- Add recursion depth/token budget guard for skill-to-skill composition.
- Add resource policy: max bytes, allowed extensions, explicit directories.
- Add skill lint step in CI (`name`, `description`, length, parseability, references).

**Risk Assessment:** MEDIUM

### PLAN 03: LangChain Agent + Multi-Provider + SSE

**Summary**
This is the critical plan and mostly correct technically, but it currently locks onto APIs that are already moving. Main risks are framework churn (`create_react_agent`), provider adapter mismatches, streaming behavior under tools, and message migration safety.

**Strengths**
- Correctly centralizes model resolution in ProviderManager.
- SSE-first API path is aligned with unified UX.
- Keeping legacy message column during migration is safer than hard cutover.
- System prompt composition approach is coherent.

**Concerns**
- **HIGH**: `create_react_agent` is deprecated in current LangGraph/LangChain references; newer guidance favors `create_agent`.
- **HIGH**: Using `ChatOpenAI(base_url=...)` for DeepSeek is now explicitly discouraged by `langchain-openai` docs (provider-specific adapters preferred).
- **HIGH**: Streaming plan depends on `astream_events(v2)` but doesn't define robust fallback behavior for message/token edge cases with tools.
- **MEDIUM**: No explicit migration/backfill/read-preference logic for dual message formats.
- **MEDIUM**: Singleton SkillLoader concurrency/reload semantics are undefined.

**Suggestions**
- Prefer `langchain.agents.create_agent` unless blocked by hard feature gap.
- Use provider-native model wrappers (or `init_chat_model`) for non-OpenAI providers.
- Define streaming fallback matrix: `messages` token stream -> `updates` step stream -> terminal response.
- Add dual-read/single-write migration policy with telemetry and cutoff criteria.
- Add SSE disconnect/cancel handling and idempotent resume semantics.

**Risk Assessment:** HIGH

### PLAN 04: Frontend Adaptation

**Summary**
As written, this plan is intentionally minimal, but it does not satisfy AGENT-04 success criteria. It's a staging plan, not a completion plan.

**Strengths**
- Low-change approach reduces immediate blast radius.
- Explicit deprecation markers can help transition communication.
- Adds skill listing endpoint surface for UI discovery.

**Concerns**
- **HIGH**: "All AI operations via `/agent/chat`" is not achieved; canvas still uses legacy invoke flow.
- **HIGH**: Creates architecture split-brain in frontend (two execution paths).
- **MEDIUM**: No migration timeline/flag strategy for progressive rollout.
- **LOW**: TODO-only marker may be forgotten without enforcement.

**Suggestions**
- Re-scope as Phase 12.1A (compat) and add 12.1B (full frontend migration) explicitly.
- Add feature-flagged adapter layer so canvas calls can route to agent path incrementally.
- Add telemetry dashboard for endpoint usage (`/skills/invoke` vs `/agent/chat`) and enforce sunset gate.
- Add E2E tests for canvas node flows under agent route before removing legacy APIs.

**Risk Assessment:** HIGH

---

### PLAN 05: Cleanup (Remove PydanticAI + Old Handlers)

**Summary**
Cleanup intent is good, but timing is risky given Plan 04 still preserves legacy canvas invocation. Deletion needs stricter dependency proofs and phased hardening.

**Strengths**
- Explicit file deletion list and import scan approach is disciplined.
- Recognizes need to keep certain canvas-critical handlers during transition.
- Includes startup + test verification step.

**Concerns**
- **HIGH**: Cleanup may race ahead of frontend/backend migration completeness and break remaining legacy flows.
- **MEDIUM**: Removing direct provider deps because they're "transitive" is brittle for runtime/import stability.
- **MEDIUM**: "Fix broken imports" is broad; no explicit compatibility shim strategy.
- **LOW**: Test plan scope not clearly including streaming, SSE reconnect, and migration-read paths.

**Suggestions**
- Gate deletions behind "zero traffic on legacy endpoints" + E2E pass criteria.
- Keep explicit direct deps for providers actually instantiated in your code.
- Add temporary compatibility module for old import paths with deprecation warnings.
- Add migration test matrix: old history -> new agent, new history -> old clients, mixed threads.

**Risk Assessment:** MEDIUM-HIGH

## Consensus Summary

*(Single reviewer — Codex. Key themes extracted below.)*

### Top Concerns (Priority Order)

1. **Plan 03 API Choices (HIGH):** `create_react_agent` may be deprecated; `ChatOpenAI(base_url=...)` for DeepSeek is now discouraged. Need to verify latest LangGraph stable API surface before execution.

2. **Plan 04 Incomplete AGENT-04 (HIGH):** Frontend adaptation is labeled as satisfying AGENT-04 ("all AI ops via agent chat") but canvas still uses legacy `skillsApi.invoke`. This is a split-brain state, not goal completion.

3. **17 Tools Near Reliability Cliff (HIGH):** Tool count is at the known LLM tool-selection degradation threshold. No gating/routing strategy defined.

4. **Plan 05 Premature Cleanup Risk (MEDIUM-HIGH):** Deleting PydanticAI + old handlers while canvas still depends on legacy invoke path creates breakage risk.

5. **Streaming Fallback Undefined (HIGH):** `astream_events(v2)` is the sole streaming strategy with no defined fallback matrix for tool-call edge cases.

### Agreed Strengths

- Clean tool layering (read/write/AI/skill) with per-call DB sessions
- Progressive disclosure via SKILL.md three-level loading is well-designed
- ProviderManager centralization for LLM resolution is correct
- Keeping legacy message column during format migration is prudent
- Disciplined file deletion list with import scanning

### Actionable Recommendations

1. **Verify `create_react_agent` vs `create_agent` status** before Plan 03 execution
2. **Re-scope Plan 04** as transitional bridge; add explicit full-migration plan
3. **Add tool gating** to keep exposed tool count ≤12-14 per context
4. **Gate Plan 05 deletions** behind E2E canvas node verification
5. **Define streaming fallback matrix** for SSE reliability
