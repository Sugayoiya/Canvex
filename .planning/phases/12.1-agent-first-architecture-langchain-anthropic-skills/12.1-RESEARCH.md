# Phase 12.1: Agent-First Architecture: LangChain + Anthropic Skills - Research

**Researched:** 2026-04-02
**Domain:** LangChain Agent Framework Migration + Anthropic SKILL.md Convention
**Confidence:** HIGH

## Summary

Phase 12.1 replaces the current PydanticAI-based agent system with LangChain/LangGraph's `create_react_agent` pattern, converts 14 existing skills into either LangChain `@tool` pure functions or declarative SKILL.md directories, extends `ProviderManager` with a `resolve_langchain_llm()` method for multi-provider LLM switching, and integrates LangSmith tracing via environment variables.

The current codebase has **14 registered skills** (across 8 categories: text, extract, script, storyboard, visual, video, canvas, asset) and **4 context query tools** (`get_project_characters`, `get_project_scenes`, `get_script_content`, `get_canvas_state`). The migration follows a clear pattern: 8 "reasoning" skills (that embed LLM calls + prompts) become SKILL.md instruction sets, 2 AI service skills (`generate_image`, `generate_video`) become `@tool` wrappers, 2 pure-function skills (`canvas.get_state`, `asset.get_project_info`) become `@tool` directly, and 2 generic LLM skills (`text.llm_generate`, `text.refine`) are deleted (Agent IS the LLM).

**Primary recommendation:** Use `create_react_agent` from `langgraph.prebuilt` with `prompt` parameter for system messages; use `agent.astream(stream_mode="messages")` for token-level SSE streaming (with fallback to `astream_events` if regression in current version); use `AsyncSessionLocal` context manager for DB access inside `@tool` functions; use Celery `.apply_async()` + polling for long-running image/video generation inside `@tool`.

### Pattern 1: LangChain @tool with DB Session (Recommended: AsyncSessionLocal)

**What:** `@tool` functions need DB access but LangChain tools don't participate in FastAPI DI. Use `AsyncSessionLocal` context manager.

**Why not InjectedToolArg:** LangChain's `InjectedToolArg` / `ToolRuntime` has known bugs as of March 2026 (issue #35931) where `_DirectlyInjectedToolArg` isn't properly filtered from Pydantic schemas. The simpler pattern of using `AsyncSessionLocal` directly is reliable and matches the existing pattern in `asset.get_project_info`.

**Example:**
```python
from langchain_core.tools import tool
from pydantic import BaseModel, Field

class GetCharactersInput(BaseModel):
    project_id: str = Field(description="Project ID to fetch characters for")

@tool(args_schema=GetCharactersInput)
async def get_characters(project_id: str) -> str:
    """Get all characters in the specified project with their names and descriptions."""
    from app.core.database import AsyncSessionLocal
    from app.models.character import Character
    
    async with AsyncSessionLocal() as db:
        result = await db.execute(
            select(Character).where(
                Character.project_id == project_id,
                Character.is_deleted == False,
            )
        )
        chars = result.scalars().all()
        return json.dumps([
            {"id": c.id, "name": c.name, "description": c.description or ""}
            for c in chars
        ], ensure_ascii=False)
```

**Security note:** The current `context_tools.py` enforces project_id ownership via `RunContext[AgentDeps]`. In the new model, the Agent already has `project_id` in context — tools validate by receiving it as an explicit parameter. The system prompt constrains the Agent to only pass the session's project_id.

### Pattern 2: Celery Async Tasks inside @tool (generate_image, generate_video)

**What:** Image/video generation takes 10-60 seconds. Cannot block the agent loop.

**Recommended approach:** Submit via Celery `.apply_async()`, then poll with exponential backoff inside the tool. Return result or timeout error.

```python
@tool(args_schema=GenerateImageInput)
async def generate_image(prompt: str, aspect_ratio: str = "16:9") -> str:
    """Generate an image using AI from the given prompt."""
    from app.services.ai.provider_manager import get_provider_manager
    
    pm = get_provider_manager()
    provider, _owner, key_id = await pm.get_provider(
        "gemini", team_id=_get_team_id(), user_id=_get_user_id(),
    )
    result = await provider.generate_image(prompt, aspect_ratio=aspect_ratio)
    return json.dumps(result, ensure_ascii=False)
```

**Alternative for truly long tasks:** If Celery is required (> 30s), the tool can submit and return a task_id, then the Agent can use a separate `poll_task` tool to check status. This avoids blocking the agent loop.

### Pattern 3: SkillLoader Three-Level Progressive Disclosure

**What:** Load SKILL.md in stages to keep context window efficient.

```python
class SkillLoader:
    def __init__(self, skills_dir: Path):
        self.skills_dir = skills_dir
        self._metadata: dict[str, SkillMeta] = {}  # L1 cache
    
    def load_metadata(self) -> list[SkillMeta]:
        """L1: Parse YAML frontmatter from all SKILL.md files at startup."""
        for skill_dir in self.skills_dir.iterdir():
            skill_md = skill_dir / "SKILL.md"
            if skill_md.exists():
                meta = self._parse_frontmatter(skill_md)
                self._metadata[meta.name] = meta
        return list(self._metadata.values())
    
    def build_system_prompt_fragment(self) -> str:
        """Inject L1 metadata into system prompt."""
        lines = ["## Available Skills\n"]
        for m in self._metadata.values():
            lines.append(f"- **{m.name}**: {m.description}")
        return "\n".join(lines)
    
    def read_skill(self, name: str) -> str:
        """L2: Return full SKILL.md body (Markdown after frontmatter)."""
        skill_md = self.skills_dir / name / "SKILL.md"
        if not skill_md.exists():
            return f"Skill '{name}' not found."
        content = skill_md.read_text()
        # Strip YAML frontmatter
        if content.startswith("---"):
            end = content.index("---", 3)
            return content[end + 3:].strip()
        return content
    
    def read_resource(self, name: str, filename: str) -> str:
        """L3: Return content of a specific resource file."""
        resource_path = self.skills_dir / name / filename
        if not resource_path.exists():
            return f"Resource '{filename}' not found in skill '{name}'."
        return resource_path.read_text()
```

### Pattern 4: SSE Streaming from create_react_agent

**What:** Map LangGraph streaming events to existing `sse_protocol.py` format.

**Recommended streaming approach:** Use `agent.astream(input, stream_mode="messages")` which yields `(message_chunk, metadata)` tuples. Map these to existing SSE events.

**Fallback approach (if `messages` mode regresses):** Use `agent.astream_events(input, version="v2")` and filter events:

```python
async def stream_agent(agent, input_messages, config):
    async for event in agent.astream_events(input_messages, version="v2", config=config):
        kind = event["event"]
        if kind == "on_chat_model_stream":
            chunk = event["data"]["chunk"]
            if chunk.content:
                yield sse_token(chunk.content)
        elif kind == "on_tool_start":
            yield sse_tool_call(event["name"], event["data"].get("input", {}), event["run_id"])
        elif kind == "on_tool_end":
            yield sse_tool_result(event["name"], str(event["data"].get("output", "")), event["run_id"])
```

**IMPORTANT:** Set `streaming=True` on the LLM constructor for token-level streaming to work.

### Pattern 5: resolve_langchain_llm Implementation

```python
async def resolve_langchain_llm(
    self,
    provider_name: str,
    model_name: str,
    *,
    team_id: str | None = None,
    user_id: str | None = None,
    **kwargs,
) -> "BaseChatModel":
    """Resolve a LangChain BaseChatModel via DB-backed credential chain."""
    api_key, _owner, _key_id = await self.resolve_api_key(
        provider_name, team_id=team_id, user_id=user_id,
    )
    if provider_name == "gemini":
        from langchain_google_genai import ChatGoogleGenerativeAI
        return ChatGoogleGenerativeAI(
            model=model_name, api_key=api_key, streaming=True, **kwargs,
        )
    elif provider_name in ("openai", "deepseek"):
        from langchain_openai import ChatOpenAI
        extra = {}
        if provider_name == "deepseek":
            extra["base_url"] = "https://api.deepseek.com/v1"
        return ChatOpenAI(
            model=model_name, api_key=api_key, streaming=True, **extra, **kwargs,
        )
    raise ValueError(f"Unsupported provider for LangChain: {provider_name}")
```

### Pattern 6: SKILL.md Format (Anthropic Agent Skills Spec)

```markdown
---
name: extract-characters
description: >
  Extract character list from screenplay or story text. Use when the user asks
  to analyze characters, extract character information, or create a character
  list from a script. Returns structured character data (name, description,
  gender, age, personality).
---

# Extract Characters

## When to Use
- User asks to "提取角色", "分析人物", "角色列表"
- A script or story text is available via `get_script` tool
- User wants structured character data from raw text

## Workflow
1. Call `get_script(episode_id)` to retrieve the screenplay text
2. Analyze the text and identify all characters
3. For each character, extract: name, description, gender, age, personality
4. Call `save_characters(project_id, characters)` to persist results
5. Report the extracted character list to the user

## Output Format
Return a JSON array where each item has:
- `name`: Character name (string, required)
- `description`: Physical appearance description
- `gender`: 男/女/未知
- `age`: Age or age range
- `personality`: Character personality traits

## Tips
- Focus on characters with dialogue or significant actions
- Infer gender/age from context clues when not explicitly stated
- Keep descriptions concise (1-2 sentences)
```

### Anti-Patterns to Avoid
- **Embedding LLM reasoning inside @tool:** Tools are pure functions. The Agent does ALL reasoning. Tools only read/write data or call external services.
- **Passing large JSON between tool calls:** Use tool arguments, not message passing. The Agent maintains conversation context.
- **Using InjectedToolArg for DB sessions:** Known buggy in langchain-core 1.2.x. Use `AsyncSessionLocal` instead.
- **Blocking the event loop in @tool:** Image/video generation must either be async-native or use Celery submit + poll.
- **Keeping PydanticAI message serialization:** The `pydantic_ai_messages_json` column in `AgentMessage` model must be replaced with LangChain-compatible serialization (use `messages_to_dict` / `messages_from_dict` from `langchain_core.messages`).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| ReAct agent loop | Custom while loop with tool dispatch | `create_react_agent` from `langgraph.prebuilt` | Handles tool binding, retries, message management, state graph |
| Multi-provider LLM | Switch statements with raw API calls | `BaseChatModel` polymorphism via `resolve_langchain_llm` | Uniform interface, streaming support, LangSmith auto-tracing |
| Observability/tracing | Custom logging interceptors | LangSmith via env vars | Zero-code integration, full call chain visibility |
| Tool input validation | Manual dict parsing in tool body | Pydantic `BaseModel` as `args_schema` on `@tool` | LLM sees JSON schema, runtime validates automatically |
| Message serialization | Custom JSON format | `langchain_core.messages.messages_to_dict` / `messages_from_dict` | Handles all message types (Human, AI, Tool, System) with metadata |
| YAML frontmatter parsing | Regex or custom parser | `yaml` stdlib + string split on `---` | Robust, handles edge cases, matches agentskills.io spec |

## Common Pitfalls

### Pitfall 1: Streaming Regression in LangGraph
**What goes wrong:** `astream(stream_mode="messages")` may not emit token-level chunks when tools are invoked (known issue in LangGraph >=0.5.0, #5249).
**Why it happens:** Internal regression in tool node handling within the prebuilt agent.
**How to avoid:** Test with the exact version being installed. Have `astream_events(version="v2")` as fallback. Filter events by `on_chat_model_stream` kind.
**Warning signs:** All tokens arrive at once after tool calls complete instead of streaming incrementally.

### Pitfall 2: DB Session Lifecycle in @tool
**What goes wrong:** Using a single shared DB session across multiple tool calls causes stale reads or concurrent modification errors.
**Why it happens:** Tools execute in the agent loop which may have long-running conversations; a session opened at request start becomes stale.
**How to avoid:** Always use `async with AsyncSessionLocal() as db:` inside each tool call — short-lived sessions per operation.
**Warning signs:** `sqlalchemy.exc.InvalidRequestError` or stale data reads.

### Pitfall 3: PydanticAI Removal Breaking Transitive Imports
**What goes wrong:** Removing `pydantic-ai-slim` may break imports in files that don't directly import it but depend on it transitively.
**Why it happens:** 43+ files import from `pydantic_ai` or `app.agent` or `app.skills` which reference PydanticAI types.
**How to avoid:** Map ALL imports before removal. Files affected:
- `api/app/agent/agent_service.py` — `from pydantic_ai import Agent`, `ModelMessagesTypeAdapter`
- `api/app/agent/context_tools.py` — `from pydantic_ai import RunContext`, `FunctionToolset`
- `api/app/agent/skill_toolset.py` — `from pydantic_ai.tools import ToolDefinition, RunContext`
- `api/app/agent/pipeline_tools.py` — `from pydantic_ai import RunContext`, `FunctionToolset`
- `api/app/api/v1/agent.py` — `from pydantic_ai import Agent as AgentCls`
- `api/app/models/agent_session.py` — No direct PydanticAI import (safe)
- Test files: `conftest.py`, `test_agent_api.py`, `test_skill_registration.py`
**Warning signs:** `ModuleNotFoundError: No module named 'pydantic_ai'` at startup.

### Pitfall 4: Message History Incompatibility
**What goes wrong:** Existing sessions have `pydantic_ai_messages_json` serialized with PydanticAI's `ModelMessagesTypeAdapter`. New agent expects LangChain message format.
**Why it happens:** Serialization format is framework-specific.
**How to avoid:** 
1. Add new column `langchain_messages_json` to `AgentMessage` (or repurpose the existing column)
2. On migration, existing sessions with PydanticAI format can be treated as "legacy" — new messages use LangChain format
3. `load_message_history` returns empty for sessions with only PydanticAI data (graceful degradation)
**Warning signs:** JSON deserialization errors when loading old conversation history.

### Pitfall 5: Tool Count Limit
**What goes wrong:** Agent becomes less reliable with too many tools (>15-20).
**Why it happens:** LLM tool selection degrades with large tool sets.
**How to avoid:** The `read_skill` and `read_resource` tools act as a "meta-tool" layer — Agent reads skill instructions dynamically instead of having 20+ tools. Keep the directly-exposed tool count to ~14-16.
**Warning signs:** Agent picks wrong tools, hallucinates tool names, or fails to call tools at all.

### Pitfall 6: create_react_agent System Prompt
**What goes wrong:** Using deprecated `state_modifier` parameter causes errors.
**Why it happens:** `state_modifier` was removed from `create_react_agent` in favor of `prompt` parameter (PR #4439, May 2025).
**How to avoid:** Use `prompt` parameter (string or SystemMessage) to set system prompt:
```python
agent = create_react_agent(
    llm,
    tools=tools,
    prompt=system_prompt_string,
)
```
**Warning signs:** `TypeError: create_react_agent() got an unexpected keyword argument 'state_modifier'`.

## Code Examples

### Complete @tool with DB Access and Error Handling
```python
from langchain_core.tools import tool
from pydantic import BaseModel, Field
import json, logging

logger = logging.getLogger(__name__)

class GetScriptInput(BaseModel):
    episode_id: str = Field(description="Episode ID to fetch script for")

@tool(args_schema=GetScriptInput)
async def get_script(episode_id: str) -> str:
    """Retrieve the screenplay/script content for a specific episode.
    
    Returns the script text content or an error message if not found.
    """
    from app.core.database import AsyncSessionLocal
    from app.models.script import Script
    from app.models.episode import Episode
    
    try:
        async with AsyncSessionLocal() as db:
            episode = await db.get(Episode, episode_id)
            if not episode:
                return json.dumps({"error": "剧集不存在"}, ensure_ascii=False)
            
            result = await db.execute(
                select(Script).where(Script.episode_id == episode_id)
            )
            script = result.scalar_one_or_none()
            if not script:
                return json.dumps({"error": "该剧集暂无剧本"}, ensure_ascii=False)
            
            return json.dumps({
                "episode_id": episode_id,
                "content": script.content or "",
            }, ensure_ascii=False)
    except Exception as exc:
        logger.warning("get_script failed: %s", exc)
        return json.dumps({"error": "无法加载剧本内容"}, ensure_ascii=False)
```

### SSE Streaming Integration with FastAPI
```python
from sse_starlette.sse import EventSourceResponse

@router.post("/chat/{session_id}")
async def chat(session_id: str, body: ChatRequest, user=Depends(get_current_user)):
    agent = await agent_service.create_agent(...)
    
    async def event_generator():
        yield sse_thinking("analyzing")
        
        input_messages = {"messages": [HumanMessage(content=body.message)]}
        
        async for event in agent.astream_events(input_messages, version="v2"):
            kind = event["event"]
            if kind == "on_chat_model_stream":
                chunk = event["data"]["chunk"]
                if hasattr(chunk, "content") and chunk.content:
                    yield sse_token(chunk.content)
            elif kind == "on_tool_start":
                yield sse_tool_call(
                    event["name"], event["data"].get("input", {}), event["run_id"]
                )
            elif kind == "on_tool_end":
                output = event["data"].get("output", "")
                yield sse_tool_result(event["name"], str(output)[:200], event["run_id"])
        
        yield sse_done(collected_text, usage)
    
    return EventSourceResponse(event_generator())
```

## Open Questions

1. **Message history format migration**
   - What we know: PydanticAI uses `ModelMessagesTypeAdapter` for serialization; LangChain uses `messages_to_dict`
   - What's unclear: Whether existing session history should be migrated or abandoned
   - Recommendation: Add `langchain_messages_json` column, keep `pydantic_ai_messages_json` for legacy reads, new messages use LangChain format. Old sessions gracefully degrade (empty history on first new message).

2. **Tool context injection (project_id, user_id, team_id)**
   - What we know: LangChain `@tool` doesn't have FastAPI-style DI; `InjectedToolArg` is buggy
   - What's unclear: Best pattern for passing session context to tools
   - Recommendation: Use Python `contextvars.ContextVar` to store session context at request start; tools read from contextvars. This is concurrency-safe and matches the existing `_current_key_id_var` pattern in `provider_manager.py`.

3. **Celery task handling in @tool**
   - What we know: `generate_image` and `generate_video` are currently `async_celery` skills
   - What's unclear: Whether to keep Celery or call provider directly in async
   - Recommendation: Call provider directly with `await provider.generate_image()` inside the @tool (which already runs async). Only use Celery if the agent loop timeout is shorter than generation time. The current implementation in `generate_image.py` already calls `provider.generate_image()` directly — the Celery wrapper is at the SkillRegistry level, not in the handler itself.

4. **SkillRegistry fate**
   - What we know: 14 skills registered via `register_all.py` → `SkillRegistry`
   - What's unclear: Whether `SkillRegistry` is still needed after migration
   - Recommendation: Keep `SkillRegistry` in reduced scope for admin listing/stats; the agent no longer invokes through it. Alternatively, the admin skill listing can read from SKILL.md metadata directly.
