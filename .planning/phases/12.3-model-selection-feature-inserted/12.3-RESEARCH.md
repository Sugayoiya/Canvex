# Phase 12.3: Model Selection Feature - Research

**Researched:** 2026-04-03
**Domain:** Full-stack model selection (Backend API + Frontend component + multi-level fallback)
**Confidence:** HIGH

## Summary

Phase 12.3 adds a complete model selection capability across the application. The backend work involves: (1) a new `GET /api/v1/models/available` endpoint that returns only models with active keys, (2) a `resolve_provider_for_model()` function using `ModelPricing` JOIN to auto-derive provider from model name, (3) a `resolve_model_for_task()` implementing a 4-layer fallback chain (manual → project default → personal/team default → system default), and (4) new `settings` JSON columns on `User` and `Team` models plus a `SystemSetting` model for system defaults.

The frontend work involves: (1) a reusable `ModelSelector` component (Pill + Popover pattern), (2) integration into AI Chat popup and Canvas AI Generate Panel, (3) default model configuration sections on project detail, personal settings, and team settings pages, (4) LLM Node simplification (removing legacy provider select), and (5) `SkillContext.model_name` injection to eliminate hardcoded providers in skill handlers.

**Primary recommendation:** Build backend-first (available models API → resolve functions → settings endpoints → SkillContext injection), then frontend (ModelSelector component → Chat/Canvas integration → settings pages → fallback toast). The `ModelPricing` JOIN is the key architectural pivot — it replaces the need for clients to know or pass `provider_name`.

### Pattern 1: ModelPricing JOIN for Provider Resolution

**What:** Instead of requiring clients to know `provider_name`, derive it from `model_name` via DB join.
**When to use:** Every time backend needs to create a provider instance from just a `model_name`.

```python
async def resolve_provider_for_model(
    model_name: str,
    *,
    team_id: str | None = None,
    user_id: str | None = None,
    db: AsyncSession | None = None,
) -> tuple[str, str]:
    """Given model_name, return (provider_name, provider_config_id) via ModelPricing JOIN."""
    stmt = (
        select(ModelPricing.provider, AIProviderConfig.provider_name, AIProviderConfig.id)
        .join(AIProviderConfig, ModelPricing.provider_config_id == AIProviderConfig.id)
        .join(AIModelConfig, ModelPricing.model_config_id == AIModelConfig.id)
        .where(
            AIModelConfig.model_name == model_name,
            ModelPricing.is_active == True,
            AIProviderConfig.is_enabled == True,
        )
    )
    # Returns provider_name for credential chain resolution
```

**Why this works:** `ModelPricing` already serves as the provider-model association table (established in Phase 12.2, PROV-03). Each `ModelPricing` row links a `model_config_id` to a `provider_config_id`, making this a natural JOIN path.

### Pattern 2: 4-Layer Fallback Chain

**What:** Cascading model resolution with graceful degradation.
**When to use:** Every AI call that doesn't have an explicit model selection.

```python
async def resolve_model_for_task(
    model_name: str | None,
    model_type: str = "llm",
    *,
    project_id: str | None = None,
    user_id: str | None = None,
    team_id: str | None = None,
    db: AsyncSession,
) -> str:
    """
    4-layer fallback:
    1. model_name (explicit user selection)
    2. Project.settings.default_{type}_model
    3. User.settings.default_{type}_model (personal) OR Team.settings (team project)
    4. SystemSetting default_{type}_model
    Raises ValueError if no model found at any layer.
    """
```

**Key design decision:** Layer 3 branches on `Project.owner_type` — personal projects check `User.settings`, team projects check `Team.settings`. This aligns with the existing ownership model.

### Pattern 4: SystemSetting KV Store

**What:** Simple key-value table for system-wide settings.
**When to use:** System default model configuration (D-15).

```python
class SystemSetting(Base):
```

### Anti-Patterns to Avoid

- **Passing `provider_name` from frontend:** D-05 explicitly deprecates this. Frontend only sends `model_name`; backend derives provider.
- **Hardcoding provider in skill handlers:** Current `generate_image.py` and `generate_video.py` hardcode `"gemini"` — this must be replaced with `resolve_provider_for_model(ctx.model_name)`.
- **Storing model selection in URL params:** Use Zustand stores (chat: persisted, canvas: session-scoped) — not URL state.
- **N+1 queries in available models API:** Must use JOINs to check for active keys in a single query, not loop through models.

## Common Pitfalls

### Pitfall 1: N+1 Query in Available Models API
**What goes wrong:** Querying each model individually to check if it has an active provider key.
**Why it happens:** The relationship chain is `AIModelConfig → ModelPricing → AIProviderConfig → AIProviderKey`, 4 tables deep.
**How to avoid:** Use a single query with JOINs: `SELECT DISTINCT model_config.* FROM ai_model_configs JOIN model_pricing JOIN ai_provider_configs JOIN ai_provider_keys WHERE key.is_active = true AND config.is_enabled = true`.
**Warning signs:** Slow `/models/available` response, many DB queries in logs.

### Pitfall 2: Race Condition in Fallback Chain
**What goes wrong:** Model becomes unavailable between `resolve_model_for_task()` and actual API call.
**Why it happens:** Key health changes (429/5xx) between resolution and execution.
**How to avoid:** The existing `get_provider_with_retry()` already handles this with key rotation. The fallback chain only resolves **which model** to use, not which key — key rotation handles the rest.
**Warning signs:** Fallback toast appearing but then actual call also failing.

### Pitfall 3: Stale Model List in Frontend
**What goes wrong:** User sees models in the selector that are no longer available (key deleted/disabled).
**Why it happens:** Model list cached too aggressively.
**How to avoid:** Use `staleTime: 5 * 60 * 1000` (5 min) for model list — balances freshness with performance. Models don't change frequently. Add `queryClient.invalidateQueries(["models", "available"])` when admin changes provider config.
**Warning signs:** User selects model, gets fallback toast every time.

### Pitfall 4: SessionCreateRequest Breaking Change
**What goes wrong:** Existing sessions created with `model_name: "gemini-2.5-flash"` and `provider: "gemini"` stop working.
**Why it happens:** Changing defaults from explicit values to `None`.
**How to avoid:** Keep `provider` field as Optional with deprecation notice. Make `model_name` default to `None` (triggers fallback). Existing sessions keep their stored `model_name`/`provider` — only NEW sessions with `None` trigger fallback.
**Warning signs:** Existing chat sessions stop working after upgrade.

### Pitfall 5: Circular Import in resolve_model_for_task
**What goes wrong:** `provider_manager.py` imports from `models/project.py`, `models/user.py`, etc.
**Why it happens:** `resolve_model_for_task()` needs to read `Project.settings`, `User.settings`, `Team.settings`.
**How to avoid:** Use lazy imports inside the function body (already the pattern in `provider_manager.py` — see `_auto_select()` method which does `from app.models.ai_provider_config import AIProviderConfig` inside the function).
**Warning signs:** `ImportError` on startup.

### Pitfall 6: ProviderIcon Location
**What goes wrong:** `ProviderIcon` is in `web/src/components/admin/` — importing from `common/model-selector.tsx` creates a cross-module dependency (common → admin).
**Why it happens:** `ProviderIcon` was created in Phase 12.2 for admin pages.
**How to avoid:** Move `ProviderIcon` from `admin/provider-icon.tsx` to `common/provider-icon.tsx` (or `ui/provider-icon.tsx`). Update existing imports in `admin/provider-card.tsx`.
**Warning signs:** Unclear import path, feels architecturally wrong.

### Pitfall 7: Zustand Persist for Chat Store
**What goes wrong:** Chat store doesn't currently use Zustand `persist` middleware, but D-04 requires persisting `selectedModelName`.
**Why it happens:** `useChatStore` was created as a plain Zustand store (no persist).
**How to avoid:** Either add `persist` middleware to the entire chat store (with partialize to only persist `selectedModelName`), or store the selection separately in localStorage. Persist middleware with `partialize` is cleaner.
**Warning signs:** Model selection resets on page refresh.

### Backend: Available Models Query (Single Efficient Query)

```python
stmt = (
    select(AIModelConfig, ModelPricing, AIProviderConfig)
    .join(ModelPricing, ModelPricing.model_config_id == AIModelConfig.id)
    .join(AIProviderConfig, ModelPricing.provider_config_id == AIProviderConfig.id)
    .where(
        AIModelConfig.is_enabled == True,
        ModelPricing.is_active == True,
        AIProviderConfig.is_enabled == True,
        AIProviderConfig.keys.any(AIProviderKey.is_active == True),
    )
    .order_by(AIProviderConfig.priority.asc(), AIModelConfig.display_name.asc())
)
```

**Note:** The `keys.any()` subquery ensures only providers with at least one active key are included. This avoids showing models whose provider has all keys disabled.

### Backend: resolve_provider_for_model

```python
async def resolve_provider_for_model(
    model_name: str,
    *,
    db: AsyncSession,
) -> tuple[str, str | None]:
    """Return (provider_name, base_url) for a model_name via ModelPricing JOIN.
    
    Raises ValueError if no active pricing association found.
    """
    stmt = (
        select(AIProviderConfig.provider_name, AIProviderConfig.base_url, AIProviderConfig.default_base_url)
        .join(ModelPricing, ModelPricing.provider_config_id == AIProviderConfig.id)
        .join(AIModelConfig, ModelPricing.model_config_id == AIModelConfig.id)
        .where(
            AIModelConfig.model_name == model_name,
            ModelPricing.is_active == True,
            AIProviderConfig.is_enabled == True,
        )
        .limit(1)
    )
    row = (await db.execute(stmt)).first()
    if not row:
        raise ValueError(f"No active provider found for model '{model_name}'")
    return row[0], row[1] or row[2]  # provider_name, effective_base_url
```

### Frontend: ModelSelector Data Hook

```typescript
export function useAvailableModels(modelType?: "llm" | "image" | "all") {
  return useQuery({
    queryKey: ["models", "available"],
    queryFn: () => modelsApi.getAvailable().then(r => r.data),
    staleTime: 5 * 60 * 1000,  // 5 minutes
    select: (data) => {
      if (!modelType || modelType === "all") return [...data.llm, ...data.image];
      return data[modelType] ?? [];
    },
  });
}
```
