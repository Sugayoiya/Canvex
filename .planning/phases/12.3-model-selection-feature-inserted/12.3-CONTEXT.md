# Phase 12.3: Model Selection Feature - Context

**Gathered:** 2026-04-03
**Status:** Ready for planning

<domain>
## Phase Boundary

为用户添加完整的模型选择能力：可用模型 API、四层 fallback 链（手动选择 > 项目默认 > 个人/Team 默认 > 系统默认）、可复用 ModelSelector 组件、Agent Chat 和 Canvas 浮动面板集成、个人/Team/项目级默认模型设置。用户只选 `model_name`，后端通过 `ModelPricing` 自动推导 provider。

**不包含在此 Phase：**
- Admin 模型管理 UI 改造（12.2 已完成）
- Provider 热插拔（deferred）
- Token 预算/成本跟踪（Phase 15）
- Pipeline 参数对齐（Phase 13）

</domain>

<decisions>
## Implementation Decisions

### API 路由
- **D-01:** 新建 `GET /api/v1/models/available`，独立 router (`models.py`)，返回 `{ llm: [...], image: [...] }` 按类型分组，仅过滤有 active key 且 is_enabled 的模型，普通登录用户可访问。不复用 admin 的 `/ai-providers/models`

### ModelSelector 组件
- **D-02:** Pill + Popover 形态。Popover 内平铺列表，按 provider 排序，每行显示 `[ProviderIcon] model_display_name [价格]`，不分组。Pill 显示当前选中的 `[ProviderIcon] model_name + chevron`
- **D-03:** 支持 `size="sm" | "md"` 两种变体。`sm` 用于 Chat 输入区和 Canvas 浮动面板，`md` 用于设置页

### Agent Chat 模型切换
- **D-04:** 支持消息级模型切换，`ChatRequest` body 带 `model_name`（后端自动推导 provider）。模型选择状态存 Zustand chat store 并持久化，刷新后保留上次选择
- **D-05:** 前端只传 `model_name`，不再传 `provider`。后端 `resolve_provider_for_model()` 通过 `ModelPricing` JOIN `AIProviderConfig` 推导 provider_name。`SessionCreateRequest` 和 `ChatRequest` 中 `provider` 字段保留但标记 deprecated，`model_name` 默认改为 `None`（触发 fallback 链）
- **D-16:** Chat ModelSelector 只显示全能力（LLM + Image）模型。后端响应包含 `features` 字段，前端 Chat 场景过滤具备 `["llm", "image"]` 的模型。Agent 推理和 @tool 调用统一使用用户选中的模型
- **D-17:** Chat 弹窗中 ModelSelector 放在输入框下方左侧，和 Mic/Send 同行，使用 `size="sm"` pill 形态

### Canvas 模型选择
- **D-06:** AI Generate Panel（节点浮动面板）底栏的 "Lib Nano Pro" 替换为 ModelSelector。根据 `focusedNodeType` 自动切换 `modelType`（text→LLM, image→Image, video→Video）。Popover 内只显示模型名和价格。execute 时 params 带上选中的 `model_name`
- **D-07:** LLM Node 内部的 provider select 和独立执行按钮为遗留代码，本阶段清理移除。LLM Node 改为和其他节点一样通过浮动面板触发执行

### Fallback 行为
- **D-08:** 当用户选中的模型不可用时，后端 `resolve_model_for_task()` 自动沿 fallback 链降级，响应中附带 `used_model` 字段。前端检测到实际使用模型与请求不同时，Sonner toast 提示 "模型 X 不可用，已自动切换到 Y"
- **D-09:** Fallback 链全部耗尽时返回 HTTP 503 "无可用模型，请联系管理员配置 API Key"。Fallback 成功降级时，前端自动将实际使用的模型持久化到当前选择状态（Zustand store / node config），避免后续重复触发 fallback 提示

### Fallback 链设计
- **D-10:** 四层 fallback：用户手动选择 > 项目默认（`Project.settings`）> 个人默认/Team 默认（同级，按项目 `owner_type` 决定走个人还是 Team）> 系统默认。个人默认存 `User.settings` JSON，Team 默认存 `Team.settings` JSON

### 默认模型设置入口
- **D-11:** 三个入口：项目详情页设置项目默认、个人设置页设置个人默认、Team 设置页设置 Team 默认。完整四层 fallback
- **D-12:** 项目详情页基本信息下方内嵌 "默认模型" section，两个 ModelSelector（LLM / Image）并排。创建项目时也可选择默认模型，未选时按 fallback 链继承（个人/Team 默认 > 系统默认）

### Skill 改造
- **D-13:** model_name 通过 SkillContext 注入，不走 params。SkillContext 新增 `model_name: str | None = None` 字段。Invoke endpoint 创建 ctx 时从请求中设置 `model_name` + 补上 `team_id`。Skill handler 统一通过 `resolve_model_for_task(ctx.model_name, ctx.project_id, ctx.user_id, ctx.team_id)` 解析 provider，消除所有硬编码 provider

### 数据存储
- **D-14:** User 和 Team model 新增 `settings: JSON` 列（与 Project 一致），存储 `{ default_llm_model, default_image_model }`。通过 API endpoint（`PATCH /users/me/settings`、`PATCH /teams/{id}/settings`）读写
- **D-15:** 系统默认模型由 Admin 在设置页配置，存 `SystemSetting` 表（`default_llm_model` / `default_image_model`）。作为 fallback 链最后一层兜底

### Claude's Discretion
- ModelSelector Popover 的具体样式、动画、z-index 处理
- `staleTime` 缓存策略（模型列表不常变，可设较长）
- `resolve_model_for_task()` 的具体函数签名和放置位置
- 个人设置页 / Team 设置页中模型设置区块的具体 UI 布局
- LLM Node 清理后的简化 UI

</decisions>

<specifics>
## Specific Ideas

- 参考 Cursor 左下角的模型选择器交互——Pill + Popover，紧凑不抢空间
- Chat 使用全能力模型（LLM + Image + Video），不允许选纯文本模型
- Fallback 降级后自动持久化选择，避免用户反复看到 toast 提示
- Canvas 浮动面板根据节点类型自动切换模型列表，用户不需要理解 provider 概念
- model_name 作为调用时快照存入 SkillContext，支持用户频繁切换模型

</specifics>
