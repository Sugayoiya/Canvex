  <action>
Create `api/app/api/v1/admin_users.py` and define router:
- `router = APIRouter(prefix="/admin", tags=["admin-users"])`

Add endpoint:
- `GET /users` (effective path `/api/v1/admin/users`)

Parameters and behavior:
- `q: str | None` (search `email` or `nickname` via `ilike`)
- `status: str | None` (`active` or `banned`)
- `is_admin: bool | None`
- `sort_by: str = "created_at"` with allowlist `created_at`, `last_login_at`, `email`
- `sort_order: str = "desc"` with allowlist `asc`/`desc`
- `limit: int = Query(20, ge=1, le=100)`
- `offset: int = Query(0, ge=0)`

Security:
- Require `user=Depends(get_current_user)`
- Call `require_admin(user)` at top

Sorting implementation:
- Use a dict mapping `sort_by` string to SQLAlchemy column reference.
- **CRITICAL for cross-DB consistency:** When `sort_by == "last_login_at"`, wrap the sort expression with `nullslast()` (from `sqlalchemy import nullslast`). This field is nullable — users who have never logged in have `last_login_at = NULL`. PostgreSQL defaults to NULLS LAST for ASC but NULLS FIRST for DESC; SQLite treats NULL as smaller than any value. Using `nullslast()` explicitly ensures NULL values always appear at the end regardless of database engine or sort direction.
- Example: `order_expr = nullslast(col.desc())` or `nullslast(col.asc())` depending on `sort_order`.
- For `created_at` and `email` (non-nullable), `nullslast()` is harmless but unnecessary — apply only to `last_login_at` for clarity.

Response schema additions in `api/app/schemas/admin.py`:
- `AdminUserListItem` fields: `id`, `email`, `nickname`, `avatar`, `status`, `is_admin`, `last_login_at`, `created_at`
- `AdminUserListResponse` fields: `items: list[AdminUserListItem]`, `total`, `limit`, `offset`

Do not include `password_hash`, `refresh_token_hash`, or `refresh_token_expires` in response.

Add to `admin_users.py`:

1. `PATCH /users/{user_id}/status`
- Request schema: `AdminUserStatusUpdate` with `status: Literal["active", "banned"]`
- For `status == "banned"`:
  - set `target.status = "banned"`
  - set `target.refresh_token_hash = None`
  - set `target.refresh_token_expires = None`
- For `status == "active"`:
  - set `target.status = "active"`
- Emit audit via `record_admin_audit(...)` with:
  - `action_type="user.status.update"`
  - `target_type="user"`
  - `target_id=user_id`
  - `changes={"status": {"old": old_status, "new": new_status}}`

2. `PATCH /users/{user_id}/admin`
- Request schema: `AdminUserRoleUpdate` with `is_admin: bool`
- Guardrails:
  - If `user_id == user.id` and target `is_admin` becomes `False`: reject with 400, emit audit with `success=False`, `error_message="self_demotion_blocked"`
  - If toggling to `False`, count active admins (`is_admin=True and status="active"`); if count <= 1 and target currently admin, reject with 400, emit audit with `success=False`, `error_message="last_admin_blocked"`
- Emit audit for success and rejection:
  - success action: `action_type="user.admin.update"`, `success=True`
  - rejection action: same action type, `success=False`, include `error_message`

Return updated `AdminUserListItem` from both endpoints.

Add to `api/app/schemas/admin.py`:
- `AdminUserStatusUpdate` with `status: Literal["active", "banned"]`
- `AdminUserRoleUpdate` with `is_admin: bool`
