# Cross-AI Plan Review — Phase 03.1

## Codex Review

### Plan 01 (Wave 1): Agent Backend

#### 1. Summary
This plan addresses the most critical backend blockers for Agent Chat and is correctly scoped to Phase 03.1 goals. Mounting toolsets, injecting context, and removing dead heartbeat logic are high-impact fixes. The main risk is under-specifying context boundaries, auth checks, and failure behavior for the new query tools.

#### 2. Strengths
- Directly maps to core chat defects (tool mounting, context injection, heartbeat dead code).
- Good separation of concerns: endpoint wiring vs. context-building/tool modules.
- `context_tools.py` as a dedicated module reduces coupling and improves testability.
- Schema touchpoint (`schemas/canvas.py`) suggests contract cleanup, not ad-hoc payloads.

#### 3. Concerns
- **HIGH:** Context query tools may expose cross-project/cross-canvas data if project/canvas/session authorization is not enforced at tool layer.
- **HIGH:** No explicit fallback behavior if context loading fails (timeouts/DB errors); chat may regress to hard failures.
- **MEDIUM:** "Schema fix" is vague; incompatible schema changes could break frontend chat parsing.
- **MEDIUM:** Removing heartbeat code without checking client expectations may leave stale reconnect/status UX assumptions.
- **LOW:** Tool-call volume may increase DB load if no caps/caching are added.

#### 4. Suggestions
- Enforce project/canvas/session ownership checks inside each context tool, not only at endpoint entry.
- Define fail-open vs. fail-closed behavior: agent should still answer with degraded context when safe.
- Add strict response schema versioning or compatibility checks for chat payloads.
- Add query limits (`top_k`, max tokenized summary size, timeout budget) to avoid context overfetch.
- Add integration tests for: wrong project ID, session switch, tool timeout, empty canvas.

#### 5. Risk Assessment
**Overall: MEDIUM** — Correct scope and strong architecture direction, but security and resilience details are essential and currently under-specified.

### Plan 02 (Wave 1): Canvas Hooks

#### 1. Summary
This plan is well-targeted for shared frontend data-flow primitives and persistence reliability. Centralizing upstream aggregation and node persistence into hooks is the right move for consistency. Biggest risks are race conditions, payload typing drift, and backend sync edge cases during rapid graph edits.

#### 2. Strengths
- Reusable hooks align with "single execution backbone" and reduce per-node duplication.
- Explicit typed aggregation supports D-07/D-08 decisions.
- Persistence strategy matches D-11 (debounce + critical immediate paths).
- Includes edge deletion sync, directly addressing a listed defect.

#### 3. Concerns
- **HIGH:** Concurrency/race risks between debounced saves, blur-trigger saves, and node execution updates.
- **HIGH:** Multi-upstream array merge rules may be ambiguous (ordering, dedupe, type conflicts).
- **MEDIUM:** Edge deletion sync may fail silently under network errors, causing local/backend divergence.
- **MEDIUM:** Debounce interval may still cause data loss on tab close/navigation without flush handling.
- **LOW:** Hook abstraction may hide errors unless surfaced consistently to node UI/state.

#### 4. Suggestions
- Define deterministic upstream merge policy (stable edge order, per-type precedence, conflict handling).
- Add optimistic update + retry/backoff + conflict reconciliation for edge/node persistence APIs.
- Add `beforeunload`/route-change flush for pending debounced writes.
- Include request cancellation or revision tokens to prevent stale writes overwriting newer edits.
- Add unit tests for hook behavior under rapid typing, rapid edge create/delete, and simultaneous node updates.

#### 5. Risk Assessment
**Overall: MEDIUM-HIGH** — Good architecture, but correctness under concurrent UI events is the key risk.

### Plan 03 (Wave 1): Chat Frontend

#### 1. Summary
This is a focused, low-complexity plan that directly fixes core UX regressions (session switching and suggestion chips). It should deliver quick wins. Primary risk is store lifecycle correctness when users switch sessions rapidly or send multiple messages in short bursts.

#### 2. Strengths
- Clear mapping to two explicit defects.
- Small, targeted file set; low scope creep risk.
- Fixing suggestion chips via shared `sendMessage` path promotes behavior consistency.
- Session list + store alignment addresses historical context continuity.

#### 3. Concerns
- **MEDIUM:** Potential stale closure/race still possible if `sendMessage` is captured in outdated handlers.
- **MEDIUM:** Session switch may trigger duplicate fetches or out-of-order state updates.
- **LOW:** Missing empty/error states for failed history load can degrade perceived reliability.
- **LOW:** Suggestion chip messages may bypass input validation/transform logic if not routed identically to manual sends.

#### 4. Suggestions
- Ensure `sendMessage` uses latest store state (functional updates/selectors, not stale captured vars).
- Add request identity guards (`sessionId` + request token) to ignore stale history responses.
- Route chip actions through exact same message pipeline as text input.
- Add UX states: loading spinner per session, retry on history fetch failure.
- Add tests for rapid session switching and repeated chip clicks.

#### 5. Risk Assessment
**Overall: LOW-MEDIUM** — Scope is tight and high-value; main risks are state synchronization under rapid interactions.

### Plan 04 (Wave 2): Canvas Node Integration

#### 1. Summary
This plan is the integration hinge for delivering visible canvas correctness, and dependency on Plan 02 is appropriate. Wiring shared hooks into node implementations and writeback paths should close most remaining canvas defects. Main risks are inconsistent adoption across node types and partial execution/writeback semantics that create confusing outputs.

#### 2. Strengths
- Correctly scheduled after hook foundation (Plan 02 dependency).
- Covers all impacted node categories (LLM/ImageGen/Extract/TextInput/Output).
- Includes both inbound data flow and outbound writeback, not one-sided fixes.
- Targets user-visible behavior improvements without violating D-14/D-15 (frontend-only graph flow).

#### 3. Concerns
- **HIGH:** Inconsistent hook integration across nodes could leave behavior fragmented (some nodes typed, others ad-hoc).
- **HIGH:** Execution result writeback may overwrite user edits or prior results without version checks.
- **MEDIUM:** Output auto-aggregation rules may be unclear with multi-upstream mixed types.
- **MEDIUM:** Frontend-only execution flow may drift from backend expectations if payload contracts are not unified.
- **LOW:** Performance risk if every node recomputes upstream data too often in large graphs.

#### 4. Suggestions
- Define a shared node I/O contract (input/output schema + metadata + timestamps/revision).
- Add writeback guardrails: last-write-wins with revision token or explicit conflict resolution.
- Standardize output aggregation policy by type (`text`, `json`, `imageUrl`) with deterministic ordering.
- Memoize upstream derivations and avoid unnecessary recompute on unrelated graph changes.
- Add end-to-end tests for: chain of nodes, multi-upstream arrays, edge deletion effects, node result persistence reload.

#### 5. Risk Assessment
**Overall: MEDIUM-HIGH** — This plan is necessary and likely effective, but integration consistency and writeback conflict handling are critical for correctness.

## Consensus Summary

> Single reviewer (Codex). Cross-plan synthesis below.

### Agreed Strengths
- Plans are logically decomposed and well-aligned with the 12 defects and key decisions
- Wave ordering is sound (Plan 04 depends on Plan 02)
- Hooks-first architecture (useUpstreamData, useNodePersistence) reduces duplication
- Tight scope — no feature creep, pure bug-fix phase
- Good separation of concerns across backend/frontend modules

### Agreed Concerns
- **Security (HIGH):** Context query tools must enforce per-request project/session auth, not rely solely on endpoint-level auth
- **Data Consistency (HIGH):** Debounced persistence + execution writeback needs explicit conflict/race strategy
- **Resilience (HIGH):** No fallback behavior defined when context loading or persistence fails
- **Contract Drift (MEDIUM):** Frontend/backend schema changes need coordinated validation
- **Performance (MEDIUM):** Context loading and graph recomputation need bounded behavior (limits, caching)

### Phase-Level Risk Assessment
**Phase 03.1 overall risk: MEDIUM-HIGH.**
The plans are directionally strong and should achieve the phase goals if executed with strict attention to auth boundaries, concurrency correctness, and deterministic data-flow contracts.

### Actionable Takeaways
1. Add project/session ownership checks inside each context tool function (not just endpoint)
2. Add `beforeunload` flush for pending debounced writes
3. Define fail-open context injection (agent works with degraded context on error)
4. Add request identity guards for session history loading (ignore stale responses)
5. Add deterministic upstream merge ordering (stable edge order)
